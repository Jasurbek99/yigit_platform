- **Sheet right-click context menu — Clear cell (foundation for more)** (feat(frontend)). `SheetCell` now wraps **every** non-hidden cell with an Ant `Dropdown` (`trigger={['contextMenu']}`), replacing the browser's native right-click menu with a Sheet-owned one so future actions (Copy value, View history, …) have a home. The only item today is a danger-styled **Clear cell**, which restores a safe path to null a `date`/`datetime` (the picker's X is intentionally disabled per the earlier accidental-wipe regression — see `SheetCellEditor` `allowClear={false}`) without sending users to the Django admin. Dispatches per field type:
  - `custom_*` rows → `PATCH /shipments/{id}/custom-fields/` `{value: ''}`; cache row's `custom_fields[key]` is set to `''` in `onMutate` for instant repaint, snapshot rolled back on error.
  - Multiselect junctions (`firm_splits` / `block_sources`) → `POST .../firm-splits/` or `block-sources/` with the items array empty (server replaces all rows, so empty = delete-all); cache row's `firm_splits` / `block_sources` is set to `[]` in `onMutate` for instant repaint, snapshot rolled back on error. **No post-success sheet invalidate** — those endpoints only return `{status, count}` so the optimistic update IS the truth; refetching here would re-introduce a ~1–2 s wait on a ~1000-row season (the original "needs refresh" symptom).
  - FK fields (`country`, `customer`, `city`, `import_firm`, `variety`, `border_point`, `vehicle_responsible`) → `PATCH /shipments/{id}/` `{field: null}` via `useShipmentPatch`, plus a companion-field wipe right before the patch fires (nulls cached `country_name`/`country_code`/`country_color`/etc.) so the cell flips empty on the next render instead of waiting for the PATCH response's `reconcileFromServer` ~200–500 ms later. The reconcile step still runs and overwrites with the authoritative response.
  - Other scalar fields → plain `PATCH /shipments/{id}/` `{field: null}` via `useShipmentPatch`, optimistic+reconcile path unchanged.

  The Clear item is **disabled (greyed out)** rather than the menu being suppressed when clearing doesn't apply (already-empty, non-editable, `cargo_code`, `has_doc_advance`/`has_sales_report` computed, bool dropdowns `peregruz`/`gornushi`, reorder mode), so future items can be added without re-arranging the wrapper. Hidden gapy `—` placeholder cells are the only ones that skip the Dropdown entirely. New `sheet.clear_cell` i18n key × 3 locales. Type-check clean. (`shipment-sheet.md` updated.)

- **Sub-blocks (e.g. `OD`, `OG` under `O`) appeared as separate rows in the weekly harvest plan grid** (fix(p3)) — `GreenhouseBlock` is a self-referencing tree where parent blocks (`A`–`O`, `M5`, `M15`) are the unit of export/sales planning and sub-blocks exist only for internal block-level control. `initialize_harvest_week` in `apps/greenhouse/services.py` filtered `is_active=True` only, so it seeded `WeeklyHarvestPlan` rows for parents AND children, leaving the grid showing `O` plus `OD` plus `OG` (triple-counted in totals/truck-estimate). Filter is now `is_active=True, parent__isnull=True`. `WeeklyHarvestPlanViewSet.queryset` defensively adds `block__parent__isnull=True` so any pre-existing sub-block rows from past initializations stop reaching the API. Migration `greenhouse.0003_delete_subblock_harvest_plans` is a one-shot `RunPython` that deletes every `WeeklyHarvestPlan` whose block has a non-null parent (no-op reverse — the orphan rows had no business existing). Import command `import_harvest_plans` was already parent-only — its `BLOCK_NAME_PATTERNS` map never contained `OD`/`OG`, so the historical pollution came exclusively from `initialize_week` calls
- **`import_firm_name` always rendered as `—` even after a successful save** (fix(p3)) — `ShipmentSheetSerializer.import_firm_name` was sourced from `import_firm.name_en`, but `ImportFirm` has no `name_en` field (only `name_company` required, `name_short` optional). DRF silently returned `None` on every read, so the cell stayed blank no matter what the user picked even though the FK was being persisted correctly. Replaced with a `SerializerMethodField` that mirrors the model's `__str__` (`name_short or name_company`). All other FK display fields (country/customer/city/variety/border_point/export_firm) were verified — only import_firm had this latent bug
- **Sheet FK dropdown values flashed stale until refetch** (fix(p3)) — `import_firm`, `country`, `customer`, `city`, `variety`, `border_point` cells read `<field>_name`, but the optimistic update only wrote the FK id. After picking a new value the cell stayed on the old name (or `—`) for the duration of the round-trip. `useShipmentPatch` now accepts an `optimisticDisplay` map (cache-only, never sent over wire), and the dropdown editor passes the picked option's label so the matching `_name` updates instantly

Its `onSuccess` invalidates the sheet query and calls `close()`, which sets `editingCell=null` and unmounts the editor — so the user could never select a second block/firm.
