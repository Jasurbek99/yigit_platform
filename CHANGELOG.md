### Changed
- **Collapsed MSSQL schemas to `dbo` with flat table-name prefixes** (refactor(db)) — eliminated three open mssql-django bugs that recurred whenever a model lived in a non-`dbo` schema: `_sql_select_default_constraint_name` (issue #402, open), M2M through-table double-bracket auto-naming (covered partially by an in-tree workaround migration), and `sql_flush` against schema-qualified tables (issue #496, open). All tables now live in `dbo`; `core_*`, `export_*`, `greenhouse_*` are name prefixes only.
  - **`schema_table()` returns `f'{schema}_{table}'`** in `apps/core/db_utils.py` (one-line change). The 52 callsites on model `Meta.db_table` now produce flat names like `core_export_firms`, `export_shipments`, `export_harvest_day_entries`. SQLite-vs-MSSQL split removed (both backends now return the same value). Critical invariant preserved: `HarvestDayEntry` (in the `greenhouse` Django app) keeps the `export_` prefix because the model owns the `export.harvest_day_entries` cross-app bridge — verified in the regenerated `apps/greenhouse/migrations/0001_initial.py`.
  - **Migration history nuked + regenerated.** All 56 pre-collapse migrations across `core` / `export` / `greenhouse` were `git mv`ed to `backend/_pre_collapse_backup/<app>/` for reference; `python manage.py makemigrations` regenerated three fresh `0001_initial.py` files containing only ORM `CreateModel` / `AddIndex` / `AddConstraint` ops — zero `RunSQL`, zero `cursor.execute`, zero `[schema].[table]` literals, zero `CREATE SCHEMA` statements. All `db_collation='Cyrillic_General_CI_AS'` declarations and all `Meta.constraints` (CHECK + UNIQUE) survive on the field/model level and were verified in the regen output. Dropped the `_is_sqlite()` helper in `db_utils.py`.
  - **6 data-seed migrations re-created** with explicit cross-app dependencies and `DJANGO_TESTING=true` guards (skip during tests so test `setUp` methods that create their own reference rows don't hit UNIQUE conflicts): `core/0002_seed_shipment_option_types`, `core/0003_seed_tomato_varieties`, `core/0004_seed_crate_types`, `core/0005_seed_greenhouse_config`, `core/0006_seed_shipment_draft_status` (moved from `export/` — model lives in core), `export/0002_seed_truck_split_defaults`. Pre-collapse data-*backfill* migrations (sheet display_order, harvest day-entry explosion, etc.) are intentionally not re-emitted because they operated on prior production state and the new DB cutover (step 7.5 of the migration plan) copies post-state directly from `YIGIT_PLATFROM`.
  - **`mssql_patches.py` deleted; custom `MSSQLSchemaTestRunner` deleted.** No more monkey-patching mssql-django's schema editor — the patched bug (`_sql_select_default_constraint_name` searching `sys.tables.name = '[core].[my_table]'`) is irrelevant for `dbo` tables. The custom test runner was incomplete schema-creation scaffolding that's no longer needed; Django's default `DiscoverRunner` handles `dbo`-only fine. `CoreConfig.ready()` no longer calls `apply_patch()`.
  - **`token_blacklist.0008_migrate_to_bigautofield` workaround** (`core/0007_pre_simplejwt_drop_blacklisted_token_unique`). simplejwt 5.3's `BlacklistedToken.token` is a `OneToOneField` → SQL Server creates an auto-named UNIQUE constraint on `token_id`. mssql-django 1.7's `_alter_field` drops the FK and the FK helper index but does *not* enumerate `sys.key_constraints`, so the auto-named UNIQUE survives and the subsequent `ALTER COLUMN token_id bigint NOT NULL` fails with `(4922) ALTER TABLE ALTER COLUMN failed because one or more objects access this column`. New migration uses `run_before = [('token_blacklist', '0008_migrate_to_bigautofield')]` to find and drop the auto-named UNIQUE constraint pre-`0008`. Fully scoped to that one column; documented removal criterion (delete when mssql-django merges a fix or simplejwt rewrites `0008`).
  - **New management command `dump_fk_map`** (`apps/core/management/commands/dump_fk_map.py`) — read-only introspection of `sys.foreign_keys` for the `core` / `export` / `greenhouse` schemas, emits `docs/MIGRATION_FK_MAP.md` with FK relationships + a topologically sorted table list. ### Added
- **Sheet Control v2 — Phase 2a backend: per-user row preferences** (feat(p3)) — users can now persist a personal row order and hide individual rows in the Shipment Sheet, independently of admin display_order and is_visible settings.
  - **New model `UserSheetRowPref`** (`export_user_sheet_row_pref`): one row per `(user, SheetRowSetting)` pair. `position INT NULL` (sparse, step 1024 — NULL inherits admin `display_order`); `is_hidden BOOL` (AND-composes with admin `is_visible`). UniqueConstraint `uq_user_sheet_row_pref`. No JSONField per ADR-0008. Migration `0035_user_sheet_row_pref` (depends on `0034`, additive only, no backfill).
  - **New endpoints** `GET/PATCH /api/v1/export/user/sheet-preferences/` — read or bulk-update current user's row order + hidden rows. PATCH is partial (absent key = no-op). Fully replaces each dimension when present (e.g. `hidden_rows: []` unhides all). Validates all submitted row_ids against `SheetRowSetting.objects.active()`; unknown/soft-deleted ids return 400 `{error: "unknown_row_ids", ids: [...]}`. Wrapped in `transaction.atomic()`. Uses `update_or_create` per row (≤50 rows — documented trade-off vs bulk_create). `IsAuthenticated` only; all querysets hard-scoped to `request.user`.
  - **`/sheet/` order resolution updated** (master plan §3.6 / ADR-0003): loads `UserSheetRowPref` in 1 extra query (never N+1); computes `effective_order = user.position ?? admin.display_order` and `effective_visible = admin.is_visible AND NOT user.is_hidden`; rows and row_settings are now served in effective_order (previously `DEFAULT_SHEET_ROWS` insertion order). New `user_preferences: {row_order: [...], hidden_rows: [...]}` top-level payload key so frontend knows the user's current state without a second API call.
  - **Tests** (`tests_user_preferences.py`, 18 tests, all green): GET empty, PATCH order (positions 1024/2048/3072), PATCH hide/unhide, combined PATCH, unknown row_id validation, soft-deleted row_id validation, user isolation, `/sheet/` payload honors user order and hide, admin is_visible=False stays hidden regardless of user prefs, query-budget assertion (≤20 total, adding 3 prefs ≤ baseline+1).
- **Sheet Control v2 — Phase 1 foundation** (feat(p3)) — full rewrite of the Shipment Sheet admin tab and the per-row permission model. Replaces the v1 single-`triggered_role` XOR model with multi-role triggers, multi-user grants, an explicit lock toggle, multi-language labels and tooltips, per-row style, soft-delete, and optimistic locking. Drops the "inactive triggered_user = lock" hack.
  - **Three new ADRs locked the design.** `ADR-0008` (no JSONField on MSSQL — split columns or child tables), `ADR-0009` (additive migrations, do not delete `0026/0028`), `ADR-0010` (explicit `is_locked` field; data-migration converts inactive-user-as-lock rows). Builds on existing ADRs 0001/0002/0006/0007.
  - **Models.** `SheetRowSetting` extended with `display_order` (sparse, step 1024), `is_visible`, `is_locked`, `label_tk/_ru/_en` and `description_tk/_ru/_en` (CharField with `Cyrillic_General_CI_AS`), `style_width/_align/_color`, `version` (auto-bumped in `save()`), `deleted_at` / `deleted_by`. New `SheetRowSettingManager` exposes `.active()` and `.visible()`. New child models `SheetRowRoleTrigger(row, role)` (replaces single `triggered_role`) and `SheetRowUserPermission(row, user, can_edit, deleted_at, ...)` with partial UQ `(row, user) WHERE deleted_at IS NULL` (MSSQL filtered index).
  - **Migrations.** Three additive migrations on top of the existing v1 schema. `0031_sheet_control_v2_base` adds the new columns, alters `triggered_user` semantics, runs two data steps (`backfill_display_order = row_number * 1024`, `migrate_inactive_user_locks` — sets `is_locked=True, triggered_user=None` for rows whose triggered_user was inactive). `0032_sheet_row_role_trigger` creates the role-trigger child table, copies existing `triggered_role` values into it, drops the old XOR check constraint, drops the column. `0033_sheet_row_user_permission` creates the user-permission child table. All `bulk_create`/`bulk_update` carry `batch_size=500`.
  - **Permissions rewrite.** `apps/core/permissions.py::can_edit_sheet_field` and `get_sheet_edit_map` rewritten for ADR-0001 + ADR-0010 semantics: superuser/admin/director immune; hidden rows blocked; `is_locked=True` requires user match OR role-trigger match OR active extra-user grant (each AND'd with field perm); soft-deleted user grants ignored. Query budget for `get_sheet_edit_map` is now ≤4 with `prefetch_related('role_triggers', 'user_permissions')` (was ≤2 in v1) — the threshold is documented in the docstring and the test asserts the new bound.
  - **Admin endpoints (URL switched from `field_key` to numeric `id`).** `GET/PATCH /admin/sheet-rows/{id}/` (PATCH requires `version` body field, returns 409 with `current_version` on mismatch — version is mandatory, not optional, per ADR-0006). `POST /admin/sheet-rows/reorder/` body `{order: [id, ...]}` updates a single row's `display_order` to the average of its neighbours; rebalances all rows when gap < 2 (ADR-0007). `POST /admin/sheet-rows/permissions/bulk/` body `{row_id, grants[], revokes[]}` is idempotent (revokes set `deleted_at`; grants `update_or_create` with `deleted_at=None`). `DELETE /admin/sheet-rows/{id}/` soft-deletes and rejects with 400 unless `is_visible=False` for ≥30 days. `POST /admin/sheet-rows/{id}/restore/` clears `deleted_at`. `?include_deleted=1` shows soft-deleted rows. AuditLog continues to record per-changed-field rows; reorder writes a single summary AuditLog entry.
  - **`/sheet/` payload — Sheet Control v2 compact format.** New top-level keys: `users_index` (Record<user_id, {name, role}>), `current_user_id`, `current_user_lang`. `row_settings[field_key]` now carries `is_locked`, `is_visible`, nested `labels {tk,ru,en}`, `description {tk,ru,en}`, `style {width?, align?, color?}`, `triggered_user_id`, `triggered_roles[]`, `extra_user_ids[]`, `can_current_user_edit`, `version`, `settings_updated_at`, `settings_updated_by_id`. Hidden rows are filtered from BOTH `rows` and `row_settings` so the frontend never iterates them.
  - **Frontend admin tab rewrite.** `SheetRowsTab.tsx` rewritten (~545 lines) with inline-edit columns for labels (3 langs), tooltip popover for descriptions, visibility/lock `Switch`es, style popover (width slider 50–500, align radio, color picker), multi-role `Select`, multi-user `Select` wired through `useBulkPermissions` (diffs old vs new). 409 conflict triggers a `Modal.confirm` with a single Refresh action. Drag-reorder posts to the new sparse-update endpoint with optimistic cache. Tier 1/2/3 plate above the table (master plan §5).
  - **Frontend Quick View.** `ShipmentSheet.tsx` reads `row_settings[fk]` from the new payload. Lock icon (`LockOutlined`) renders next to row label when `is_locked=true && can_current_user_edit=false`. Labels prefer `row_settings[fk].labels[current_user_lang]`, falling back to the existing i18n `t(label_key)` mechanism. `style.width / align / color` applied to cells when defined. **Per-row `style.width` in this column-major layout is a known visual mismatch** (column header stays at `COL_WIDTH_SHIPMENT`); admins should leave it unset for now — flagged as Sprint 2 cleanup along with line-count split of `SheetRowsTab.tsx` and the systemic `IRowConfig` snake_case vs `sheetRowConfig.ts` camelCase pre-existing TS debt.
  - **Hooks.** New / rewritten: `useSheetRowSettings`, `useSaveSheetRowSetting` (typed 409 error, refresh modal), `useReorderSheetRows`, `useBulkPermissions`, `useSoftDeleteSheetRow`, `useRestoreSheetRow`. All keyed by numeric `id`, all use the existing httpOnly-cookie axios instance.
  - **Mock data.** `mock/shipmentSheet.ts` updated to the v2 payload (locked row, role-triggered row, extra-user-granted row).
  - **i18n.** New keys in tk / ru / en for the rewritten admin tab columns, popovers, modals, lock tooltip, and the Tier 1/2/3 plate.
  - **Out of scope (deferred).** Per-user row order + IndexedDB cache + BroadcastChannel multi-tab sync (Phase 2). `Shipment.is_archived` + auto-archive cron + Archive view (Phase 3). Stuck shipments dashboard with SLA escalation (Phase 4).

### Fixed
- **`tests_sheet_settings_admin.py` regressions from Phase 2a version-locking** (fix(p3)) — 7 PATCH tests written for Phase 1 started failing with `400 {'error': 'version is required for updates.'}` after the Phase 2a rewrite of `views_sheet_settings.py` made the `version` body field mandatory (was optional in Phase 1). All PATCH bodies now supply `'version': row['version']`; sequential double-PATCHes in the same test use the bumped version from the first response. (Confirmed Phase 2a regression via `git diff da40347 -- views_sheet_settings.py`.)
- **`tests_shipment_sheet.py` pre-existing test bugs** (fix(p3)) — three tests added in the `da40347` commit were silently broken before Phase 2a work began:
  - `test_firm_splits_auto_fill_official_kg` and `test_firm_splits_auto_fill_official_kg_three_firms` assumed `TruckSplitDefault` rows for `num_firms=2` (9,000 kg) and `num_firms=3` (6,000 kg) were seeded in `setUpTestData`, but no seed call existed. Without seed rows, `get_default_truck_weight(N)` falls back to `18100/N` (9,050 and 6,033.33), so both tests were returning the wrong expected values. Fixed by adding `TruckSplitDefault.objects.get_or_create()` + `invalidate_truck_split_cache()` in `setUpTestData`.
  - `test_block_sources_explicit_weight_honored` compared sorted string weights (`sorted(['12000.00', '6500.00'])`) lexicographically — string `'1'` < `'6'` so `'12000.00'` sorted before `'6500.00'`, reversing the expected order. Fixed by converting to floats before sorting.
  - (Confirmed pre-existing via `git worktree` checkout of `da40347` — all three fail identically at that commit with no Phase 2a involvement.)

### Added
- **Forecast layer for Weekly Harvest Plan** (feat(p3)) — daily-grain harvest data: each block-day cell now carries Plan, Forecast, and Actual values with full per-cell audit (timestamps, submitters, submission-window state, revision counts). KPI computation is deferred but every audit field needed to compute KPIs later is captured from day one. See **ADR-017**.
  - **`HarvestDayEntry` (daily grain) replaces 12 wide columns on `WeeklyHarvestPlan`.** New table at `export.harvest_day_entries` (one row per `weekly_plan × entry_date`) holding `plan_value` / `forecast_value` / `actual_value` plus per-layer audit columns (`*_submitted_at`, `*_submitted_by`, `plan_state`, `forecast_window`, `forecast_revision_count`, `actual_finalized_at`, `actual_source`) plus `last_override_at/by/reason` snapshot. Empty-vs-zero distinction: `value IS NULL` = "not entered" (em-dash), `value = 0 AND *_submitted_at IS NOT NULL` = explicit confirmed zero (italic). `weekday` allows 0–6 so end-of-season Sunday harvesting is supported (admin-entered, not bitmask-gated). Migration `greenhouse.0004_harvestdayentry_harvestdispatchlog_and_more` is atomic: creates new tables, backfills 6 daily rows per existing weekly plan (1,998 rows from 333 plans, preserving "never entered" via the zero-AND-no-submission rule → 257 NULL/1,741 valid plan_values), drops the wide columns + approval fields, drops FK constraints + indexes for the dropped FKs via `RunSQL` (`DF_whp_status` default constraint also dropped — works around the mssql-django default-constraint-drop bug), adds `locked_at`.
  - **Approval workflow REMOVED — submission is final** (per user direction). `WeeklyHarvestPlan.status` / `approved_at` / `approved_by` / `rejected_at` / `rejected_by` / `rejection_note` columns dropped. `submit_harvest_plan` / `approve_harvest_plan` / `reject_harvest_plan` services removed. `submit` / `approve` / `reject` / `bulk-submit` / `bulk-approve` / `bulk-reject` viewset actions removed. Replaced by `submit_weekly_plan(weekly_plan, user)` which records `submitted_at` / `submitted_by` and computes `plan_state` (`on_time` / `late` / `critical_late`) per HarvestDayEntry from current time vs config-driven deadlines.
  - **Tighter permissions, AD-15-aligned.** New role constants in `apps/core/roles.py`: `HARVEST_DAY_WRITE = {'admin', 'greenhouse_manager', 'warehouse_chief'}`, `HARVEST_DAY_OVERRIDE = {'admin'}` (admin-only with required `reason` text). `PLAN_WRITE` / `PLAN_APPROVE` removed (no callers after approval-flow drop). Service-layer enforcement: greenhouse_manager can edit Plan + Forecast (primary window) for their own assigned blocks; warehouse_chief can edit Forecast (fallback / same-day windows) and Actual on any block; admin can edit anything anytime with required `reason` that writes to `AuditLog` and `last_override_*` snapshot; export_manager / director are view-only.
  - **`GreenhouseConfig` singleton + `OperatingDayException`.** New `core.greenhouse_config` table (one row, `pk=1` enforced via `save()` override) for admin-tunable values: `plan_deadline_weekday` (default 4=Friday), `plan_late_until_weekday` (6=Sunday), `plan_critical_late_at_weekday` + `_at_time` (Monday 00:00), `forecast_primary_open` / `_close` (17:00/18:00 day-before), `forecast_fallback_close` (09:00 day-of), `forecast_same_day_close` (23:59), `notification_lead_minutes` (60), `truck_capacity_kg` (18500 — was hardcoded in frontend), `operating_days_bitmask` (0b0111111 = Mon–Sat by default; bits 0–6 = Mon–Sun), `timezone_name` (Asia/Ashgabat). `OperatingDayException(date UNIQUE, is_holiday, note)` for ad-hoc holiday calendar — dispatcher skips notifications/auto-tasks on `is_holiday=True` days.
  - **Time-based dispatcher (mgmt command + system cron, NOT Celery).** `apps/greenhouse/dispatcher.py` exposes pure `evaluate_triggers(now_local, config) -> list[TriggerEvent]` and `fire(event)` with idempotency via `HarvestDispatchLog(trigger_kind, target_user, scope_date)` UNIQUE. Run via `python manage.py run_harvest_dispatcher` from system cron every 5 min. **No SMS** in this iteration — in-app notifications only. **No Celery / Redis** — chosen mgmt-command + cron after comparing Celery+Redis / Django-Q2 / APScheduler (zero new deps, idempotent by DB UNIQUE, ≤90 trigger events/day doesn't justify a queue; revisit Celery when pallet rollup arrives). Cron entry documented in `docs/operations/cron.md` (Linux + Windows Task Scheduler).
  - **Personal kanban auto-tasks: TODO no-op.** Dispatcher includes `# TODO: evaluate_kanban_rules(event)` at the no-op call site. - **Sub-blocks (e.g. `OD`, `OG` under `O`) appeared as separate rows in the weekly harvest plan grid** (fix(p3)) — `GreenhouseBlock` is a self-referencing tree where parent blocks (`A`–`O`, `M5`, `M15`) are the unit of export/sales planning and sub-blocks exist only for internal block-level control. `initialize_harvest_week` in `apps/greenhouse/services.py` filtered `is_active=True` only, so it seeded `WeeklyHarvestPlan` rows for parents AND children, leaving the grid showing `O` plus `OD` plus `OG` (triple-counted in totals/truck-estimate). Filter is now `is_active=True, parent__isnull=True`. `WeeklyHarvestPlanViewSet.queryset` defensively adds `block__parent__isnull=True` so any pre-existing sub-block rows from past initializations stop reaching the API. Migration `greenhouse.0003_delete_subblock_harvest_plans` is a one-shot `RunPython` that deletes every `WeeklyHarvestPlan` whose block has a non-null parent (no-op reverse — the orphan rows had no business existing). Import command `import_harvest_plans` was already parent-only — its `BLOCK_NAME_PATTERNS` map never contained `OD`/`OG`, so the historical pollution came exclusively from `initialize_week` calls
- **`import_firm_name` always rendered as `—` even after a successful save** (fix(p3)) — `ShipmentSheetSerializer.import_firm_name` was sourced from `import_firm.name_en`, but `ImportFirm` has no `name_en` field (only `name_company` required, `name_short` optional). - **Sheet FK dropdown values flashed stale until refetch** (fix(p3)) — `import_firm`, `country`, `customer`, `city`, `variety`, `border_point` cells read `<field>_name`, but the optimistic update only wrote the FK id.
